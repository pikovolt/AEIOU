## 1. ドメイン知識：タイムシートとは何か



### 1.1 タイムシートの基本



アニメ制作に用いられるタイムシートとは、アニメーションのタイミングを書き込んだ表形式の指示書である。



- 行方向：時間軸

- 列方向：セルの区分（A, B, C, D, E...）



フィルム撮影基準の 24fps（1秒間に24コマ）で、シート1枚が6秒分。1秒、半秒（0.5秒）、1/4秒の位置に基準線が引かれる。
※シートの作りはスタジオによって異なる可能性あり。



### 1.2 ツールが解決する課題



タイムシートに書かれた動画部分のタイミング情報を、After Effects のキーフレーム情報（Time Remap）に変換する。



手作業では、セル番号からフレーム数と秒数を計算し、AE上で1つずつキーフレームを打つ必要がある。AEIOUはこの変換をセル入力→クリップボードコピーで自動化する。



### 1.3 変換の計算式



```

タイムリマップの秒数 = (セル番号 - 開始番号) ÷ FPS

```



- FPS：24 または 30

- 開始番号：AEのプロジェクト設定に依存（アニメ撮影では通常「1」開始）

- 有効桁：6桁（AEの Keyframe Data フォーマット仕様）

- C#実装では `t.ToString("g6")` で文字列化



### 1.4 Keyframe Data フォーマット



AEにペーストするテキスト形式の例：



```
Adobe After Effects 4.0 Keyframe Data

    Units Per Second	24
    Source Width	640
    Source Height	480
    Source Pixel Aspect Ratio	1
    Comp Pixel Aspect Ratio	1

Time Remap
    Frame	seconds
    0	0
    3	0.0416667
    6	0.0833333
    9	0.125
    12	0.166667
    15	0.208333
    18	0.25
    21	0.291667

End of Keyframe Data
```



`Frame seconds` に続く数列が「AE上のフレーム数」と「タイムリマップの秒数」の対。フレーム数はコマ数であり秒数ではない。タイムリマップの秒数は、レイヤーが参照するコンポジションの fps 基準で計算される。ツールに設定した fps とコンポジションの fps が合っていない場合、意図しない結果になる。



ヘッダーのバージョン番号（`4.0` 等）は AE のバージョンによって異なる。AEIOU では `aeiou.ini` 内の `remapVersion` タグで変更可能。



### 1.5 特殊なセル概念



**カラセル**：「セルがない」ことを示す。タイムシート上ではマス一杯に☓印を付ける。ツール上は `.`（ドット）キーで入力。実装上はカラセルに「値」を設定しており、セル枚数以上の長さのコンポジションを用意し、十分に大きな値を指定することでカラセルとして機能する（コンポジション必須の仕様）。



**継続線（縦の棒線）**：「動いてはいないが、セルが表示されている」ことを示す。セルの入力がある場合に、次のカラセルが出現するまで縦棒を描画する。



**イロハニホヘト指定**：実際のタイムシートでは数値以外にイロハニホヘト（またはアイウエオ）の指示が使われる場合がある。テンキーで良い対応方法がないため、AEIOUでは非対応。タイミング修正で番号振り直しを省いた書き方が選択された場合などに見られる。



### 1.6 切り貼りと中抜き



タイムシートが手書きの場合、丸ごと書き直すのは時間が掛かるため、部分的な修正（物理的な切り貼り）が好まれる。



- **切り貼り（追加範囲）**：タイムシートに範囲を追加する操作。AEIOUではピンク色で表示。

- **中抜き（無効範囲）**：タイムシートの一部を無効にする操作。AEIOUでは灰色で表示。「中抜き」はAEIOU特有の表現の可能性がある。



この機能がない場合、大幅修正時にすべてのフレームの追加・削除を手計算する必要がある。

## 2. 設計判断：なぜこう作ったか



### 2.1 設計コンセプト



AEIOUは UIフィールがすべてのツールである。基本コンセプトは以下の通り。



**見た目**：なるべくタイムシートの見た目に合わせる。ただし描画処理でレスポンスが落ちない程度に留める。必要十分なサイズで、AEの横に常に出しておける形にする。



**色使い**：タイムシートの見た目合わせ。可能であればカスタマイズできることが望ましい。描画処理によるレスポンス低下を避ける範囲で。



**応答性**：とにかく軽快に打てること。打ち込み中に遅滞してテンポが崩されないこと。



**操作性**：メインはひたすら数値入力なので、テンキーベースの操作が基本。テンキーだけで完結するのが理想。



**ロバスト性**：アプリが落ちてデータが消えないこと。基本シンプルな処理だけにして、とにかく落ちないようにする。最悪でもコピペでデータを取り出して仕切り直しできるようにする。



**普遍性**：シート打ちに必要十分かつ最小限の機能に特化させる。シンプル・イズ・ベスト。




### 2.2 入出力仕様



**入力形式**：

- ツール上のシートへの手入力

- After Effects の Keyframe Data

- STS フォーマット（バイナリ形式）



**出力形式**：

- After Effects のテキスト形式キーフレームデータ（クリップボードにコピー）

- 複数レイヤーへの同時出力に対応



**対応AEバージョン**：AE 4.x 以降の Keyframe Data フォーマット



### 2.3 性能要件



- 10000行のタイムシートデータを1秒以内に変換

- デフォルトではシート13枚半分の尺、セル5つ分（A〜E）

- Undo は可能な限り無制限（上限65535回）。Redo は未実装。

- 64bit対応（.NET Framework ベース）



### 2.4 Undo あり・Redo なしの判断



Undo は追加したが、Redo は実装していない。1から書き直しの際に追加した機能であり、Redo の必要性は限定的と判断された。

## 3. 操作体系：キー入力の設計



### 3.1 操作の4パターン



操作はほぼ以下の4パターンに集約される：



1. 数値の入力

2. 入力の削除

3. 範囲の選択

4. 範囲のコピー・カット・ペースト



### 3.2 数値入力



テンキーの `0`〜`9` で各マスに数値を入力する。入力中は桁が追加されていく（例：`1`, `2`, `3` と押すと `123` になる）。



### 3.3 数値入力中の編集



- `BS`（BackSpace）：入力中は1桁削除（例：`123` → `12`）

- `Enter`（Return）：入力を確定し、次のマスに進む。選択範囲単位で移動する。



### 3.4 自動改行（コマ打ち幅の維持）



数値入力の前に、事前に縦に複数のマスを選択しておくと、Enter時に選択マス単位でカーソルが移動する。



2コマ打ち、3コマ打ちを行う場合がほとんどなので、この機能は必須。例えば3コマ選択状態で入力→Enter→3行下に移動、を繰り返す。



### 3.5 BackSpace（非入力中）



数値入力をしていない場合のBackSpaceは、1つ前のタイミングの内容を削除する。



- 選択範囲内に値がある場合：範囲内の値が消える

- 空欄が続く場合：一番近いキーフレームを探して削除する

- 基本は選択範囲ずつカーソル移動する



### 3.6 自動数値入力



テンキーの `+` / `-` で連番入力を行う。`+` を連打すると `1, 2, 3, 4, 5...` と各マスにカウントアップしながら値が入力される。



### 3.7 範囲拡張・縮小



テンキーの `*` / `/` で選択範囲を1行ずつ増減する。



### 3.8 カラセル入力



`.`（ドット）キーでカラセル（☓印）を入力する。

## 4. 実装構造：コードの読み方



### 4.1 全体構造



Form1.cs のデザイン画面は、DataGridView と TextBox を置いているだけ。キー入力のイベントハンドラ内で処理を振り分けている。



**意図的にオブジェクト指向的な設計構造を排除している。** 理由は以下の通り：



- 日々の作業の手隙に実装しているため、その場で簡単に結果を確認できる実装に限定した

- オブジェクト指向的な構造だと、設計資料を残しても自分で後から見た時にすぐにコード修正ができなかった

- 使うとしても、土台になる機構（作った後に変更しない部分）に限定



コピー・ペーストも、グローバル変数にバッファを用意して直接操作するレベルの実装。逆に言えば、この部分は使われ方を知らないと修正不能な箇所でもある。



### 4.2 キーイベント処理



DataGridView の `dataGridView1_KeyDown` イベントハンドラで、キーコードを元に動作を決定する。



- `KeyDown`：実際の処理を行う

- `KeyPress`、`KeyUp`：設定はあるが「なにもしない」実装



処理の流れ（BackSpaceの場合）：

1. キーコード判定

2. 専用関数 `deleteRect_with_backspace` を呼び出し

3. イベントの処理済みフラグを立てて終了



### 4.3 ステート管理



ステート表は作成していない。動かしながら必要な状態を適宜 ON/OFF する方式。



**基本の状態変数：**



| 状態 | 説明 |

|------|------|

| 選択範囲 | DataGridView の選択中マスの範囲。DGVの状態を反映する場合と、動作結果として更新される場合がある |

| コピー範囲 | コピー・カットした値の範囲をバッファに記憶 |

| マウス押下座標 | マウス操作イベント時のカーソル座標 |

| ドラッグ中フラグ | マウスダウン時に確定、ボタンリリースまたは初期化時に false |

| 初回編集フラグ | 起動直後は true、キー入力が行われると false。キー入力中かどうかの判定に使用 |



**初回編集フラグによる動作分岐の例（BackSpace）：**



- `true`（入力中）：1文字削除

- `false`（非入力中）：範囲内の値の削除、なければ前に遡って直近の入力値の削除



**バッファ・リスト：**



| データ | 説明 |

|--------|------|

| コピーバッファ | コピー・カット操作時に選択範囲の値の内容を記録。コピー範囲は別途記録（バッファ記憶のオブジェクト実装ではない） |

| 中抜き範囲リスト | 無効範囲の開始・終了行を収めた Range クラスのリスト |

| 切り貼り範囲リスト | 追加範囲の開始・終了行を収めた Range クラスのリスト |

## 5. Python実装サンプルとの対応



GitHub上の PySide6 サンプル（GridView）は、上記 C# 実装のコアロジックを300行未満で再現したものである。



| C# (AEIOU) | Python (GridView) | 対応する概念 |

|-------------|-------------------|-------------|

| DataGridView | QTableWidget | グリッドUI |

| `dataGridView1_KeyDown` | `keyPressEvent` + `key_handlers` 辞書 | キーディスパッチ |

| 初回編集フラグ | `is_editing` | 入力中/非入力中の状態管理 |

| 選択範囲の記憶 | `shift_base_row` + `selectedRanges()` | 範囲選択の管理 |

| `deleteRect_with_backspace` | `handle_backspace_key` | BS動作の分岐 |

| 「AEへコピー」 | `handle_ctrl_o_key` → `calc_timing` → `copy_timing` | タイムリマップ変換+クリップボード |

| `EditMode` 無効化 | `setEditTriggers(NoEditTriggers)` | グリッド標準編集の無効化 |



Python サンプルに含まれていない AEIOU の機能：Undo、STS形式の入出力、切り貼り/中抜き、自動数値入力（`+`/`-`）、カラセル、継続線描画、.ini設定、ドラッグ操作。

## 6. 引き継ぎ時の注意事項



### 6.1 リファクタリングについて



4000行超の Form1.cs を見て「クラス分割すべき」と判断するのは自然だが、現在の構造は意図的な選択である（4.1節参照）。リファクタリングを行う場合は、以下を理解した上で判断すること：



- テンキー操作の応答性が最優先事項である

- グローバル変数によるバッファ管理は、コードの見通しと引き換えに即座の動作確認を可能にしている

- 分割する場合は、ステート変数の依存関係を先に把握する必要がある



### 6.2 バージョン・環境



- C# / .NET Framework / Visual Studio Community でビルド

- 64bit対応（PC の 64bitOS 化に伴い v3 で対応）

- v3 より前のバージョンのサポートは不可（当時のコンパイラが有償で、コードも手元にない）

- GitHub Actions によるビルドパイプラインあり



### 6.3 関連リソース



- リポジトリ：https://github.com/pikovolt/AEIOU

- Python実装サンプル：セル入力のコアロジック再現（PySide6 / QTableWidget）

- STS形式の仕様：「タイムシート STS形式」で検索
